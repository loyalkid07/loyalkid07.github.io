1000 Pages, 8 Hours, and a Doom Loop

What really happens when you throw threads at the GIL — and the three wrong moves I made before I actually fixed it.

8 HOURS
SYSTEM UNDER LOAD

01 The Setup

Picture a pipeline that takes in scanned, multi-page PDF documents, turns each page into an image, cleans that image up, and then sends it to an OCR model to extract the text. The OCR step was already running concurrently, and it worked well — a page came in, an async call went out to the OCR model, and while it waited on the network, the event loop went and started the next page. Clean, fast, no drama.

So when I got to the preprocessing step — the OpenCV stage before OCR even sees an image, doing adaptive resizing, contrast enhancement, denoising — the logic seemed obvious. Parallelism had just worked for one stage of this pipeline. Why wouldn't it work for the next one too? I reached for the same instinct: throw it on a thread pool, let asyncio.gather fan it out, and let it run.

with ThreadPoolExecutor(max_workers=max_preprocessing_workers) as executor:
    preprocessed_pages = await asyncio.gather(
        *[loop.run_in_executor(executor, preprocess_page, page) for page in page_images]
    )

On small test files this looked completely fine. A few pages, a couple seconds saved, nothing alarming. I moved on, half-aware that OpenCV's C++ calls release the GIL while they run, which I took as evidence that threading the whole stage was basically safe. I hadn't actually thought through how much of the work — the NumPy array handling, the encode/decode, the orchestration code itself — never leaves Python at all, and never releases anything.

02 The Nightmare

Then we started benchmarking against real volume — large, multi-hundred-page scanned documents, the kind this pipeline actually has to deal with in production, not clean test fixtures.

It fell apart. Not "slow" — falling apart. Activities timing out. The workflow engine retrying work that hadn't actually failed. Long stretches where the workflow just sat there, stalled, doing nothing visible at all. I sat through a full run on an 1,100-page document just to see it through to the end: almost 8 hours, most of it not real processing time but the pipeline tripping over itself.

My first move was the obvious one: bump max_preprocessing_workers higher. More workers, more parallelism, right? It didn't help. If anything it made things worse — more threads fighting over the same lock isn't more throughput, it's more traffic at the same intersection.

Second move: add more retries on the activity running both preprocessing and OCR. This felt reasonable at the time — activities were timing out, so give them more chances to succeed. I didn't yet understand that I was feeding the exact thing that was breaking everything: every retry threw more preprocessing work at a process that was already starved and struggling, which meant more contention, which meant more timeouts, which meant more retries. I was pouring fuel on a fire I couldn't see yet.

Third move, and I'll just say it plainly: I disabled preprocessing entirely. Not an engineering decision — a relief valve. I'd been watching the same benchmark fail for long enough that I needed it to just stop, so I could think without a retry storm running in the background. It worked, in the sense that the pain stopped. It didn't fix anything.

None of the three touched what was actually wrong. They were all reactions to a symptom I hadn't named yet. At some point I stopped trying to make the failure go away and started trying to understand why it was happening at all — and that's the part that actually mattered.

The benchmark numbers, once I started logging them properly instead of just living through them, made the shape of the problem visible before I had a name for it:

03 Going Back to First Principles

Once I stopped patching symptoms, I had to get precise about a few things I thought I already had clean.

Process vs. thread. A process has its own memory space — nothing in it touches another process without going through an explicit channel: a pipe, a socket, shared memory you set up deliberately. A thread shares its process's memory with every other thread in it. That sharing is cheap — no copying, no serialization, instant visibility — and it's also exactly why threads can step on each other, which is exactly why CPython needs a way to stop that at the bytecode level.

The GIL. One lock, held by one thread at a time, gating execution of Python bytecode. It's a property of CPython, not of Python the language. This is the one I'd actually gotten wrong, quietly, without realizing it: I knew OpenCV's C++ calls release the GIL while they run, and I'd treated that fact as "threading this is fine." What I'd skipped over is that the GIL release only covers the C++ call itself — every bit of Python sitting around it (array slicing, dtype conversions, encode/decode, the orchestration loop) still needs the lock back the moment control returns to Python. On a small test file, that gap is invisible. At real volume, with enough threads doing enough of that glue work simultaneously, that gap is where all the contention actually lived.

CPU-bound vs. I/O-bound. Not about which library you reach for. It's about where the bottleneck physically lives. If the CPU is maxed out doing real computation, waiting doesn't help, because nothing's waiting. If the bottleneck is a socket or a disk and the CPU is idle in the gap, that idle gap is exactly what async and threads can fill — which is why threading worked fine for OCR and not for this.

The event loop. A single thread asking the OS, over and over, "which of the things I'm waiting on are ready?" — on Linux, backed by epoll. When something's ready, it resumes the coroutine waiting on it. The whole model depends on that thread never actually blocking for any real stretch of time. Block it, and the loop stops doing its one job: responding to things promptly.

04 What Was Actually Happening

GIL contention doesn't just slow down the threads fighting over it. It throttles the entire process's ability to make any Python-level progress — including code that has nothing to do with image preprocessing. The asyncio event loop lives on a thread in that same process. Running a callback, resolving a completed future, letting the workflow engine's SDK send an activity heartbeat — all of it needs the GIL too. My preprocessing threads were saturating GIL acquisition, and the event loop thread was standing in the exact same line, with no priority over it.

That's where it forks into two failure paths, both coming from the same root:

The left path is the one that's intuitive once it's laid out: starved loop, missed heartbeat, false timeout, retry, more load, repeat. The right path is the one that actually fooled me at the time, because from the outside it just looked like the OCR service being flaky. But if preprocessing upstream is stalling and bursting instead of producing a steady stream, the OCR pods see uneven, clumpy load instead of smooth throughput — exactly the kind of pattern that produces queueing and timeouts on a service sized for steady-state traffic. Both paths trace back to the same root: CPU-bound work sitting somewhere it could starve the process's ability to do anything else, including the orchestration that was supposed to keep things moving.

This is the point in digging through it where my second wrong move stopped looking like a reasonable judgment call and started looking like the thing that had been making everything worse the whole time. Adding retries to the activity wasn't a neutral attempt to ride out flakiness — it was directly feeding the left branch of this exact diagram, before I had any idea the diagram existed. The 8-hour number for an 1,100-page document was never 8 hours of real preprocessing work; preprocessing a page is cheap. It was a comparatively small amount of real work plus a doom loop's worth of retries and stalls, and that overhead doesn't scale linearly with page count — it scales with how many chances the system gets to trip into the cascade, which scales with the number of batches, which scales with the number of pages. More pages didn't just mean more work. It meant more chances for the loop to compound.

05 The Actual Fix

Bypassing the GIL, correctly

The direct swap is ProcessPoolExecutor instead of threads:

import os
from concurrent.futures import ProcessPoolExecutor
import multiprocessing as mp

def get_cpu_quota() -> int:
    try:
        with open("/sys/fs/cgroup/cpu.max") as f:
            quota, period = f.read().split()
            if quota != "max":
                return max(1, int(quota) // int(period))
    except FileNotFoundError:
        pass
    return len(os.sched_getaffinity(0))

K8S_CPU_LIMIT = int(os.getenv("K8S_CPU_LIMIT", get_cpu_quota()))

ctx = mp.get_context("spawn")
global_process_pool = ProcessPoolExecutor(
    max_workers=K8S_CPU_LIMIT,
    mp_context=ctx,
    max_tasks_per_child=200,
)

def preprocess_page(page_image) -> str:
    import cv2
    cv2.setNumThreads(0)
    # ... existing OpenCV/NumPy logic ...
    return processed_page

This is also what let preprocessing go back on. Disabling it had bought breathing room, not a solution, and turning it back on with the old ThreadPoolExecutor still sitting there would have just restarted the same nightmare on a delay. This is the version that actually stayed on.

A ProcessPoolExecutor gives each worker its own interpreter and memory space — its own GIL, completely separate from the parent's. Preprocessing one page has no dependency on any other page, so this is embarrassingly parallel work, and separate processes let it actually run in parallel across physical cores instead of taking turns on one.

Two details here mattered more than the headline swap, and both are the kind of thing that doesn't show up until you're running in a cluster, not on a laptop.

First: how the pool gets sized. The naive version sizes off os.cpu_count() — which returns the host machine's core count, not the Kubernetes pod's CPU quota. A pod limited to 4 CPUs on a 64-core node reports 64 from that call. Size the pool off it and you've quietly rebuilt the exact problem you just fixed, one layer down — 64 processes contending for 4 CPU-shares instead of threads contending for the GIL, with full process-fork-and-interpreter-init cost per worker stacked on top. It passes every local test, because a laptop's os.cpu_count() usually does match reality, and it only breaks in the cluster, which is the worst possible place to find out. Reading the cgroup quota directly, falling back to os.sched_getaffinity(0), reflects the CPU set the process is actually scheduled on.

Second: cv2.setNumThreads(0), inside each worker. Without it, OpenCV spins up its own internal thread pool inside every process, fanning out across all visible CPUs again — the same oversubscription bug one layer down, at the native-library level instead of the Python-process level. Same mistake, different floor of the building.

One more thing worth getting right: the pool uses spawn explicitly instead of the platform default. On Linux, ProcessPoolExecutor defaults to fork, which copies the parent's memory at the moment of forking — including lock state held by threads that don't get copied along with it. Forking a long-lived worker that already has an active event loop and SDK threads running is a real way to copy a half-held lock into a child with no thread that can ever release it, which deadlocks on first use. spawn costs more at startup — a fresh interpreter per worker — but the pool is a singleton paying that cost once, not per task, so it's worth it. I set this deliberately rather than learning it the hard way; it's a documented enough hazard in the multiprocessing docs that it wasn't worth finding out about in production.

With the CPU work isolated in child processes, the parent process — where the workflow worker and event loop live — does almost no CPU work of its own. It submits a task and waits on an IPC pipe, which costs it almost nothing. Heartbeats fire on schedule. And because preprocessing throughput is steady instead of bursty, the OCR pods stop seeing the uneven load that was triggering backpressure on their end too. Both branches of the cascade close, not just the one I noticed first.

What run_in_executor is actually doing

This fix only makes sense if you're precise about what loop.run_in_executor() does, because it's doing more than "run this function elsewhere."

It takes a synchronous, blocking callable and a concurrent.futures executor, submits the callable to that executor, and wraps the resulting Future in an asyncio.Future the event loop knows how to await. That's the entire bridge. The loop doesn't run your preprocessing function itself and doesn't need to know OpenCV exists — it just needs to poll a future for completion, the same way it polls a socket for readiness. "Wait for this OS event" and "wait for this executor task" look almost identical from where the loop sits.

The detail that mattered for this specific fix: calling loop.run_in_executor(None, fn, *args) tells asyncio to use its default executor — a ThreadPoolExecutor the loop owns internally. That's exactly why my original instinct felt so natural; it's the path of least resistance asyncio hands you by default. But you're not stuck with the default. Passing an explicit executor — loop.run_in_executor(global_process_pool, fn, *args) — tells the loop to submit there instead. That one argument is the whole mechanism behind this fix: the loop's offloading machinery doesn't care if the executor is threads or processes, because either way it's just awaiting a future. It's not a different API. It's the same bridge, pointed somewhere else.

It's also the cleanest way to see why CPU-bound and I/O-bound work need genuinely different handling, not just different executor choices. The OCR call, by contrast, never needed an executor at all:

async def run_ocr(page_image) -> str:
    result = await client.chat.completions.create(...)
    return result.choices[0].message.content

The HTTP client here is natively async. await suspends the coroutine, the loop goes and services other work, the response arrives, the readiness event wakes the loop, the coroutine resumes. No thread, no process, no bridge needed. Executors exist for the case where the work isn't natively async — a blocking call with no async version, or CPU-bound work with nothing to yield during at all. run_in_executor is the adapter for that case specifically. It's not a free upgrade; it only does its job when the executor behind it actually matches the kind of work you hand it. That mismatch is the whole reason this post exists.

06 The Trade-off I Accepted

This fix isn't free, and it doesn't fail the way the old code failed. A per-call ThreadPoolExecutor just gets torn down and rebuilt the next time the activity runs if something inside it dies — self-contained, self-healing. A singleton ProcessPoolExecutor doesn't have that property. If one worker process crashes — a segfault inside OpenCV's C++ extensions on a malformed image is a real way for that to happen, not a hypothetical — every subsequent submission to that pool raises BrokenProcessPool until the whole worker process restarts. One bad image can take down preprocessing for every activity scheduled on that worker, not just the one that hit it.

max_tasks_per_child=200 in the code above is the mitigation — it cycles workers periodically and bounds how much damage a slow leak or a crash can do before a fresh worker takes over. I added this going in, not after getting burned by it; I'd rather take the small, known cost of periodic worker restarts than find out the hard way what a long-lived worker accumulating memory or state does after a few thousand pages. Whether that caution was strictly necessary, I can't say for certain — I hadn't run it long enough at the time to know if it would have crashed without the cap. I'd rather be honest about that than claim a war story I don't actually have. What I can say is this trades a slower, more resilient failure mode for a faster, more fragile one, and it only pays off because it's mitigated, not because the risk disappeared.

07 The Numbers

Two separate measurements, kept apart on purpose, because they answer different questions.

The first is the full-pipeline numbers from the benchmark runs — what it actually cost end to end, retries and stalls included: 300 pages at 2.5–3 hrs, 600 pages at 4–6 hrs, 1000–1100 pages at roughly 8 hrs, across iterations. These numbers include the doom loop. They're a measurement of the failure as much as the workload.

The second is the preprocessing stage in isolation, after the fix: a 10–20 page batch went from roughly 1.5–2 minutes to under 15–20 seconds. Call it a 6x improvement, at the chunk level, specifically for preprocessing.

Preprocessing Bottleneck (10-20 Page Batch)

The connection between the two is the actual story. A 6x speedup at the preprocessing stage doesn't on its own explain an 8-hour pipeline collapsing into something dramatically shorter — preprocessing was never the majority of those 8 hours. What the fix really did was remove the trigger for the cascade. Once preprocessing stopped starving the event loop, heartbeats stopped getting missed, retries stopped getting triggered, and the doom loop that was responsible for most of the original 8 hours just stopped happening. The 8-hour number was mostly retry-storm overhead sitting on top of a small amount of real work. Fixing the per-batch bottleneck didn't just make the real work faster — it deleted almost all the work that wasn't real to begin with.

08 Scaling It Up

The fix above operates on a single batch of pages. The system needed to handle documents with 10,000+ pages, which meant composing this with a second layer of concurrency, not replacing it.

A separate batching layer chunks large documents into smaller payloads — say, 20 pages each — and uses asyncio.gather to run workflow executions for each chunk concurrently. It's the same offloading mechanism from the section above, one level up: instead of the loop awaiting a process-pool future for one image, it's awaiting workflow-execution futures for entire chunks, each of which does its own process-pool preprocessing and asyncio OCR calls underneath. Processes crunch pixels at the bottom layer; asyncio orchestrates everything above it — workflow dispatch, OCR calls, the chunk fan-out itself. Neither layer does the other's job, which is why they compose instead of fighting each other for the same resource.

09 The General Pattern, Stripped of This Specific Pipeline

None of this is really about OCR pipelines. It's about a question every concurrent system eventually forces you to answer: where does this specific piece of work actually spend its time, and does the tool I'm reaching for address that, or just rearrange it?

That question has a fairly small, reusable answer.

  • If the bottleneck is the CPU itself — numerical code, image transforms, parsing, anything where the processor is genuinely busy computing — threads in CPython don't help. They can't, structurally: the GIL means only one thread executes Python bytecode at a time, so threads on CPU-bound work buy you concurrency in name only, plus the real cost of context-switching between threads that are all waiting on the same lock. The fix here is multiprocessing — separate interpreters, separate GILs, separate memory — and it's worth that cost specifically because there's no other way to get true parallel execution out of CPython for this category of work.
  • If the bottleneck is I/O — a network call, a disk read, anything where the CPU is sitting idle waiting for bytes — both threads and asyncio work, because the GIL gets released the moment a thread blocks on a syscall, and a coroutine yields control the moment it hits an await. Either way, the CPU is free during the wait, and that's the resource being reclaimed. asyncio tends to win at higher concurrency because a coroutine is far cheaper than an OS thread, but the underlying reason either approach works at all is the same: there was idle time to fill, and you filled it.
  • The event loop's role in all of this is easy to undersell. It's a single thread, and it only does its job — accepting new work, dispatching callbacks, sending out heartbeats, whatever else it's responsible for — if it's never blocked for any meaningful stretch. That's a fragile property by design. Anything sharing that process that starves Python-level execution, even something that looks completely unrelated to the event loop's own work, degrades it. This is the part that's genuinely easy to miss: the damage isn't contained to whatever's causing the contention. It's contained to the process, and the event loop lives in that process too.
  • And the constraints you're running inside of are not always the constraints you think you have. os.cpu_count() answers "how many cores does this machine have," not "how many cores am I actually allowed to use" — and in any containerized environment, those are frequently different numbers. Code that's correct on a laptop and wrong in a cluster is one of the most common ways a concurrency fix looks finished before it actually is.

Put together, that's most of what there is to get right: diagnose where the time is actually going before picking a tool, understand that the GIL's effects aren't contained to the threads causing them, and check what you're actually allowed to use before deciding how much of it to use. The rest — ProcessPoolExecutor, run_in_executor, cgroup quotas — is just the specific vocabulary CPython gives you to act on those three things correctly.
