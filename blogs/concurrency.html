The Wrong Executor

01 / CONTEXT

The AgentXBlox medical document pipeline was taking over two minutes for ten pages. The immediate instinct was concurrency — wrap the OpenCV preprocessing in ThreadPoolExecutor and gather it with asyncio. This made things catastrophically worse inside a Temporal worker environment. The workers didn't just get slow. They started dropping requests. Temporal saw timeouts, triggered retries, and the retries made it worse — a cascading doom loop of failures and inflated latency that started from a single wrong tool choice.

02 / INSIGHT

The root cause: OpenCV preprocessing — CLAHE contrast enhancement, bilateral denoising, adaptive resizing, color-aware binarization — is heavily CPU-bound. Threads in CPython fight over the GIL. They don't achieve parallelism for CPU-heavy work; they context-switch rapidly without executing simultaneously. Inside a Temporal worker, this is catastrophic. The ThreadPoolExecutor ties up all available worker threads. The async event loop becomes starved — it can't service heartbeats, it can't process new activities. Temporal detects the missed heartbeats and activity timeouts. It triggers automatic retries. But the retries hit the same starved workers, creating a doom loop of cascading failures and insanely inflated extraction times. The system didn't degrade gracefully; it amplified the failure. Fix: ProcessPoolExecutor spins up separate OS processes — each gets its own Python interpreter, bypasses the GIL entirely, and never touches the event loop thread pool. The I/O-bound OCR API calls (asyncio.gather to the dots-ocr model) use await — control yields during the network wait, batching 10-20 pages simultaneously without blocking. Two tools, two different bottlenecks, one combined 70% latency reduction.

03 / TAKEAWAY

Know your bottleneck before reaching for a concurrency primitive. CPU-bound work — image processing, math, compression — needs ProcessPoolExecutor. It gets its own Python interpreter and bypasses the GIL. I/O-bound work — network calls, API requests, file reads — needs asyncio. The event loop yields while waiting; you get concurrency without threads. Mixing them wrong in a workflow engine context doesn't degrade gracefully. It cascades. The 70% latency drop came from choosing the right primitive, not from adding more of the wrong one.
