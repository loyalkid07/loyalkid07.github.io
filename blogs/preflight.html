The Preflight Thesis

01 / CONTEXT

LLM gateways are built on a hidden assumption: that the large model is the right place to make routing, retrieval, and generation decisions. It isn't. A model that costs $15/million tokens shouldn't be deciding whether a request needs RAG, which memory store to query, or whether it even deserves a context window at all. These are classification and selection problems. They have known, cheap solutions. The large model is a generation machine — it should only be invoked when generation is actually what's needed.

02 / INSIGHT

A fine-tuned encoder classifier runs in ~12ms and handles intent classification with high accuracy on your domain. A parallel preflight pipeline — memory fetch, vector search, budget estimation — adds another ~100ms but eliminates large-model calls on misrouted, low-value, or already-answerable requests. The bin-packing assembly step gives the large model a pre-composed, token-budget-aware context instead of raw retrieval results. The large model's job becomes: generate the best response given this already-curated context. That's it. Every other decision has been made upstream.

03 / TAKEAWAY

The question to ask before calling your LLM isn't "what should I generate?" It's "does this request actually require generation?" Most production LLM workloads have a significant fraction of requests that could be answered by the preflight system alone — because the answer is in cache, the request is too low-quality to spend compute on, or the intent classification reveals it should route to a different system entirely. The small model earns its preflight latency by making the large model's job smaller, better-scoped, and more often unnecessary.
